## Bible idea: the four tags under a word

Look again at one entry in an interlinear — say the Greek *λόγος* in John 1:1. Around it you'll find:

- a **dictionary form** — the word as a dictionary would list it (its citation form), so every
  inflected appearance points back to one headword;
- a **Strong's number** — *G3056*, a numbered handle from Strong's concordance that lets a reader
  with no Greek look the word up;
- a **parsing** — a terse code saying *noun, nominative, singular, masculine*; for a verb it'd carry
  tense, voice, mood;
- a **gloss** — a single editorial word, *"Word,"* printed under the original so you can read along.

On paper they sit in a row, easy to blur together. In the model they are four distinct things.

## App idea: four labels, four roles

<figure>
  <WordLabels />
  <figcaption>One Original Word fans out to lemma, Strong's, morphology, and gloss. Lemma and
  Strong's are separate id spaces in the re-sourceable lexicon; morphology is a raw code that
  decodes via a table; the gloss is the one label stored inline on the hub.</figcaption>
</figure>

Each label is its own idea ([ADR-0007](https://github.com/riolly/bible-codex/blob/main/docs/adr/0007-original-word-hub-is-a-morpheme-grained-externally-keyed-bridge.md) #6–#8):

- <span class="term">Lemma</span> — the **dictionary form**, and the lexicon's **spine**. It's the
  thing occurrence statistics ("this word appears 330 times") and future dictionary entries
  (BDB, Thayer) hang off. An Original Word references it by `lemma_id`.
- <span class="term">Strong's</span> — a **numbered lexical id** (G#### / H####) with its own
  dictionary entry. Crucially it lives in **its own number space**, separate from the lemma — Strong's
  lumps and splits words differently than a lemma list does. And it is a **join key only**: it
  identifies a word's lexeme but is **never an Anchor** — you never address an occurrence by its
  Strong's number, only by coordinate.
- <span class="term">Morphology</span> — the parse. The source of truth is a raw `parse_code` on the
  word (e.g. `N-NSM`); a small **decode table** keyed by that code turns it into queryable features
  (noun · nominative · singular · masculine). Not wide per-word columns — a code plus a lookup, so
  Greek cases and Hebrew stems don't fight in one sparse table.
- <span class="term">Gloss</span> — the one **editorial cue** shown under the original (*"Word"*),
  optionally with helper words. It's stored **inline on the hub**, and it is *not* a translation —
  it's a translation-agnostic reading aid, distinct from any Token's actual surface text.

<div class="callout">
  <span class="tag">The distinction to hold onto</span>
  <p><strong>Lemma ≠ Strong's.</strong> Both identify "which word," but they're two separate
  identifier spaces — a lemma is the dictionary headword (the statistics spine); a Strong's number
  is a fixed concordance id. The model keeps them as two columns (`lemma_id`, `strong_id`) precisely
  because a source can ship one without the other.</p>
</div>
